INFANT GROWTH MONITORING SYSTEM - PROJECT SUMMARY
================================================================================

[1. PROJECT OVERVIEW]

The Infant Growth Monitoring System is a comprehensive AI-powered mobile and web 
application designed to monitor infant well-being through advanced cry analysis, 
facial pain detection, and developmental tracking. The system provides real-time 
insights into infant health and behavioral patterns.

Users: Parents, pediatricians, healthcare clinicians, and researchers monitoring 
infant health and development.

Tool Type: Decision-support tool with medical-grade AI analysis capabilities. 
The system provides analysis results and confidence scores to inform parental 
decisions and clinical assessments, but does not serve as a definitive diagnostic tool.

Key Features:
- Audio cry classification (Hungry, Pain, Burping, Discomfort, Tired)
- Facial pain detection and expression analysis
- Comprehensive fusion analysis combining multiple data sources
- Growth prediction and forecasting (height/weight)
- Child behavior and development tracking
- Postpartum recovery guidance
- User profile and settings management


[2. TECH STACK]

FRONTEND:
  Framework: Expo (React Native cross-platform framework)
  Language: TypeScript
  React Version: 19.1.0
  React Native Version: 0.81.5
  Routing: Expo Router 6.0.15 (file-based routing with (tabs) layout)
  Navigation: React Navigation (bottom tabs navigation)
  State Management: React Context API (auth-context.tsx)
  Audio Processing: expo-av, react-native-a udio
  Image Handling: expo-image, expo-image-picker
  UI Components: Custom themed components, Collapsibles
  Icons: Expo Vector Icons, Expo Symbols
  Animations: react-native-reanimated, react-native-gesture-handler
  Linting: ESLint with Expo config
  Build Tool: Expo CLI with native modules

BACKEND:
  Framework: FastAPI 0.104.1
  Language: Python 3.8+
  Server: Uvicorn 0.24.0 (ASGI server)
  CORS Middleware: FastAPI built-in CORSMiddleware
  ML Inference: TensorFlow 2.20.0, Keras 3.13.2
  Traditional ML: scikit-learn 1.8.0, XGBoost 3.2.0
  Audio Processing: librosa 0.11.0, soundfile, sounddevice, audioread
  Audio Enhancement: noisereduce 3.0.3
  Computer Vision: OpenCV 4.13.0.92, MediaPipe 0.10.32
  Data Processing: Pandas 3.0.1, NumPy 1.26.4
  Model Persistence: joblib 1.5.3
  Request Handling: python-multipart 0.0.6
  Serialization: Pydantic 2.12.5

DATABASE: 
  Type: File-based storage (no traditional RDBMS)
  ML Models: Saved as .pkl files (joblib serialization)
  Configuration: .env variables for API endpoint configuration

THIRD-PARTY APIs & SDKs:
  MediaPipe: Face landmark detection and facial expression analysis
  FFmpeg: Audio file processing and encoding (Windows requirement)
  TensorFlow/Keras: Deep learning model inference
  No external APIs required for core functionality

[10. ML MODELS & PREDICTION LOGIC]

AUDIO CRY ANALYSIS - TWO-STAGE CLASSIFICATION PIPELINE

File: backEnd/routers/cry_router_audio.py
Function: predict_cry(file: UploadFile)

STAGE 1: Pain Detection (Binary Classification)
  Model File: backEnd/mlModels/Cry/model_a_pain.pkl
  Scaler File: backEnd/mlModels/Cry/scaler_a.pkl
  Input Features: 355 audio features (see below)
  Output: Binary classification (0 = No Pain, 1 = Pain)
  Model Type: Unknown (loaded via joblib, likely sklearn classifier)
  Probability: Returns probabilities [p_nopain, p_pain]
  
STAGE 2: Hunger vs Normal Classification (Binary Classification)
  Model File: backEnd/mlModels/Cry/model_b_hunger.pkl
  Scaler File: backEnd/mlModels/Cry/scaler_b.pkl
  Input Features: 355 audio features (same as Stage 1)
  Output: Binary classification (0 = Normal, 1 = Hunger)
  Model Type: Unknown (loaded via joblib, likely sklearn classifier)
  Probability: Returns probabilities [p_normal, p_hunger]
  
Decision Tree:
  if Stage1_prediction == Pain:
    final_label = "pain_cry"
    confidence = p_pain (from Stage 1)
  else:
    if Stage2_prediction == Hunger:
      final_label = "hunger_cry"
      confidence = p_hunger (from Stage 2)
    else:
      final_label = "normal_cry"
      confidence = p_normal (from Stage 2)

Feature Extraction:
  Input: Audio file (WAV, MP3, M4A, etc.) via librosa
  Duration: Fixed 5.0 seconds (padded or truncated)
  Sample Rate: 22050 Hz
  
  Preprocessing:
    1. Silence detection: Check max amplitude > 0.005
    2. Noise reduction: noisereduce library with prop_decrease=0.8
  
  Features Computed (355 total):
    1. MFCC (40 coefficients) - Mean (40 features) + Std (40 features)
    2. Mel Spectrogram (128 bands) - Mean (128) + Std (128)
    3. Chroma STFT (12 bins) - Mean (12)
    4. Spectral Contrast (7 bands) - Mean (7)
    Total: 40+40+128+128+12+7 = 355 features
  
  Normalization: StandardScaler applied before prediction

Error Handling:
  - Silent audio: Returns HTTP 400 with error message
  - Invalid format: Returns HTTP 500
  - Missing models: Returns HTTP 503

FACIAL PAIN DETECTION - RANDOM FOREST CLASSIFICATION

File: backEnd/routers/cry_router_img.py
Function: predict_face(file: UploadFile)

Model File: backEnd/mlModels/CryTranslater/saved_models/img_rf_pain_classifier3.pkl
Landmark Model: backEnd/mlModels/CryTranslater/Notebooks/face_landmarker.task
Predicts: Pain vs No Pain (binary classification)
Model Type: Random Forest Classifier
Input Features: 3 facial biomarkers (see below)
Output: Binary (0 = No Pain, 1 = Pain)
Confidence: Returns probabilities [p_nopain, p_pain]

Feature Extraction (using MediaPipe Face Landmarker):
  Input: Image file (JPG, PNG, etc.)
  Landmark Detection: 468 facial landmarks detected
  Selected Landmarks:
    - LEFT_EYE: [159, 145, 133, 33] (4 points for eye opening)
    - RIGHT_EYE: [386, 374, 362, 263] (4 points for eye opening)
    - MOUTH: [13, 14, 61, 291] (4 points for mouth opening)
    - LEFT_BROW_INNER: 55, LEFT_EYE_INNER: 133
    - RIGHT_BROW_INNER: 285, RIGHT_EYE_INNER: 362
  
  Biomarkers Computed:
    1. EAR (Eye Aspect Ratio)
       - Calculation: vertical_distance / horizontal_distance
       - Interpretation: Lower EAR indicates squinting (pain indicator)
       - Computed for both eyes, then averaged
    
    2. MAR (Mouth Aspect Ratio)
       - Calculation: vertical_distance / horizontal_distance
       - Interpretation: Higher MAR indicates mouth opening (crying)
    
    3. Brow Score
       - Calculation: (left_brow_eye_dist + right_brow_eye_dist) / (2 * eye_span)
       - Interpretation: Lower brow score indicates lowered/furrowed brows (pain)
  
  Result Vector: [ear, mar, brow_score] (3 features)

Model Output:
  - Prediction: 0 (No Pain) or 1 (Pain)
  - Probabilities: [p_nopain, p_pain]
  - Returns: label="pain_expression" (if pred==1) or "no_pain" (if pred==0)
  - Confidence: probability of predicted class (0-100%)

Error Handling:
  - No face detected: Returns HTTP 400
  - Landmark extraction failed: Returns HTTP 400
  - Model loading failed: Returns HTTP 503

FUSION ANALYSIS - MULTI-MODAL DECISION FUSION

File: backEnd/routers/cry_router_fusion.py
Function: predict_cry_reason(request: FusionPredictionRequest)

Model File: backEnd/mlModels/Cry/fusion_model_calibrated.pkl
Encoder File: backEnd/mlModels/Cry/fusion_label_encoder.pkl
Predicts: Cry reason (multi-class classification)
Model Type: Calibrated classifier (likely scikit-learn DecisionTreeClassifier or ensemble)
Input Features: 12 features (see below)
Output Classes: Hunger, Pain, Discomfort, Tiredness, Diaper, Neutral
Confidence: Probability for each class

Feature Engineering (from FusionPredictionRequest):
  Context Features:
    1. baby_age_months: float [0-36]
    2. time_since_feed_hours: float [0+]
    3. time_since_sleep_hours: float [0+]
    4. room_temperature_celsius: float [15-35]
    5. diaper_status_encoded: categorical (Clean=0, Wet=1, Soiled=2 or similar)
  
  Model Integration Features:
    6. audio_predicted_class: str (Hunger, Pain, Discomfort, Tiredness)
    7. audio_confidence: float [0-1]
    8. image_predicted_class: str (Pain, No-Pain)
    9. image_confidence: float [0-1]
  
  Derived Features:
    10. model_disagreement: bool (1 if audio != image prediction)
    11. ambiguous_case: bool (1 if either confidence < 0.7)
  
  Session/Debug:
    12. session_id: str (always "API_REQUEST" for production)

Preprocessing:
  - Convert to pandas DataFrame
  - All features properly typed
  - No scaling/normalization step shown (model handles internally)

Post-Processing:
  1. Decode prediction using LabelEncoder to get class name
  2. Get probability scores from predict_proba()
  3. Map probabilities to class names
  4. Determine confidence level:
     - Low: confidence < 0.60
     - Medium: 0.60 ≤ confidence < 0.80
     - High: confidence ≥ 0.80
  5. Generate contextual messages:
     - If time_since_feed > 3h: "It's been X hours since last feeding"
     - If time_since_sleep > 2h: "Baby hasn't slept for X hours"
     - If diaper_status != Clean: "Diaper is [status]"
     - If room_temp outside 20-28°C: "Room temperature may be uncomfortable"

Output:
  {
    "predicted_cry_reason": str (predicted class),
    "confidence": float (0-1),
    "confidence_level": str,
    "confidence_message": str,
    "context_info": str (concatenated contextual messages),
    "all_class_probabilities": dict[str, float],
    "input_summary": dict,
    "disclaimer": str
  }

Error Handling:
  - Invalid field ranges: HTTP 400 with message
  - Model prediction error: HTTP 500
  - Model not loaded: HTTP 503

### D. AUTHENTICATION FLOWS

**Flow 1: User Sign-Up**
1. App loads → Navigation guard checks session → Redirects to sign-in
2. User taps "Sign Up" → (auth)/sign-up screen
3. Fills form (full name, email, password) with validation on blur
4. Submit → signUpWithEmail() → Supabase creates auth.users record
5. Auto-trigger creates profiles record
6. Redirects to /(auth)/add-infant
7. User fills infant profile (name, DOB, gender)
8. Submit → inserts to infants table with parent_id = user.id
9. Navigation guard detects session → Redirects to /(tabs)

**Flow 2: User Sign-In**
1. Sign-in screen shows email/password form
2. Submit → signInWithEmail() → Supabase validates credentials
3. Returns session with JWT token → stored in AsyncStorage
4. Auth context fetches user profile from databases
5. Navigation guard detects session → Redirects to /(tabs)

**Flow 3: Google OAuth**
1. User taps Google button on sign-in/sign-up
2. Opens native Google Sign-In dialog (requires dev build)
3. User authenticates with Google → returns idToken
4. signInWithGoogle() exchanges idToken with Supabase
5. If new user: Auto-creates profile → redirects to /(auth)/add-infant
6. If existing user: Sessions restored → redirects to /(tabs)

**Flow 4: Password Reset**
1. Sign-in screen → "Forgot password?" link → /(auth)/forgot-password
2. User enters email → resetPasswordForEmail() sends reset email
3. Email contains link with recovery token
4. User clicks link → resets password on Supabase reset page
5. Returns to sign-in and authenticates with new password

**Flow 5: Log Out**
1. Profile screen → "Log Out" button → Confirmation alert
2. On confirm: signOut() clears session from AsyncStorage
3. Navigation guard detects !session → Redirects to /(auth)/sign-in

- Child records linked to parent accounts
- Columns: id (UUID PK), parent_id (FK to profiles), name, date_of_birth (DATE), gender (ENUM)
- RLS: Parents can SELECT/INSERT/UPDATE/DELETE own infants
- One-to-many: Multiple infants per parent
